# IS-054 — Project overview aggregation (design)

**Task:** IS-054 [BE] · Wave C — Observability & evidence · P0
**Issue:** [#75](https://github.com/AI-nclisive/iot-simulator/issues/75)
**Owning spec:** `SPEC.md` → *Observe Enabled And Running Data Sources*
**Branch:** `feat/IS-054-project-overview-aggregation`

## Problem

The frontend landing/workspace screen renders one card per project
(`ProjectSummary`) and needs per-project rollup numbers it cannot derive cheaply
today: how many data sources are configured, how many are running, and how many
reusable artifacts (recordings) exist. The
[FE↔BE contract map](../../FRONTEND_BACKEND_CONTRACT_MAP.md) maps these three
counts to IS-054 and explicitly defers true `lastActivity` to IS-055/IS-083
(runtime/activity history, not yet built). The issue title also calls for an
"attention" signal: sources that are unhealthy.

Without this endpoint the FE must fire two extra fetches per project to compute
the badges, which it currently fakes from mock stores.

## Scope

In scope:
- A single read-only aggregation endpoint returning, for **every** project, the
  three counts plus an attention count.

Out of scope (tracked elsewhere):
- True `lastActivity` / "recent" timeline — **IS-055** (runtime-event history),
  **IS-083** (user-activity audit).
- Per-source health detail — **IS-053**.
- Pagination/filtering of the collection — **IS-074**.

## Decisions

| Decision | Choice | Rationale |
| --- | --- | --- |
| Endpoint shape | New collection `GET /api/v1/projects/overview` | The landing screen renders all projects; one fetch avoids N round-trips and leaves the plain `GET /projects` CRUD contract untouched. |
| Signals | 3 counts + `sourcesNeedingAttention` | Honors the contract-map gap split; defers `lastActivity` to IS-055. |
| Attention definition | `runtimeState ∈ {ERROR, STALE}` | These are the unhealthy runtime states a user must act on; `STARTING`/`STOPPED` are normal. |
| Layering | New `ProjectOverviewService` (domain) composing existing services | Reuses the runtime-state derivation (single source of truth); keeps the web layer thin; matches the service-per-aggregate convention and ArchUnit boundaries. |
| Id field name | `projectId` | Unambiguous in an aggregation DTO distinct from the project resource; FE maps at the client edge. |
| Concurrency | None (no ETag/If-Match) | Read-only derived view, not a concurrency-controlled resource. |

## Architecture / components

- **`domain/project/ProjectOverview.java`** — immutable record (the aggregate).
- **`domain/project/ProjectOverviewService.java`** — `@Service`; depends on
  `ProjectService`, `DataSourceService`, `RecordingService`. One public method:
  `List<ProjectOverview> overview()`.
- **`api/project/ProjectOverviewController.java`** — thin `@RestController`
  mapped to `/api/v1/projects/overview`, with a nested `ProjectOverviewResponse`
  DTO + `from(ProjectOverview)` mapper. Separate from `ProjectController` so CRUD
  stays focused. Spring resolves the literal `/overview` segment ahead of the
  `/{id}` path variable, so there is no route collision.

## API contract

`GET /api/v1/projects/overview` → `200 OK`, `application/json`:

```jsonc
[
  {
    "projectId": "…",
    "name": "…",
    "configuredSources": 4,        // count of data sources in the project
    "runningSources": 2,           // runtimeState == RUNNING
    "reusableArtifacts": 7,        // count of recordings
    "sourcesNeedingAttention": 1   // runtimeState ∈ {ERROR, STALE}
  }
]
```

- No request inputs. Empty project set → `[]`.
- One row per project, preserving `ProjectService.list()` ordering.
- OpenAPI/Swagger entry is auto-generated by springdoc (IS-030); no manual doc
  wiring.

## Data flow

```
overview():
  for each p in projects.list():
    sources    = dataSources.list(p.id())          // runtimeState already derived via supervisor
    configured = sources.size()
    running    = count(sources where runtimeState == RUNNING)
    attention  = count(sources where runtimeState ∈ {ERROR, STALE})
    artifacts  = recordings.list(p.id()).size()
    emit ProjectOverview(p.id(), p.name(), configured, running, artifacts, attention)
```

`dataSources.list()` derives `runtimeState` from the runtime supervisor, so the
running/attention tallies use the same source of truth as the data-source API.

## Error handling

No inputs ⇒ no 4xx originated by the endpoint. Downstream exceptions propagate
through the existing `ProblemDetail` mapping (IS-032).

## Testing

- **`ProjectOverviewServiceTest`** (domain unit): stub/fake the three services and
  verify
  - configured = all sources;
  - running counts only `RUNNING`;
  - attention counts `ERROR` **and** `STALE` (and excludes `STARTING`/`STOPPED`/`RUNNING`);
  - reusableArtifacts = recordings count;
  - multiple projects each tallied independently;
  - empty-projects case → empty list.
- **Controller web test**: `GET /api/v1/projects/overview` returns `200` and the
  exact JSON field set; follows the repo's existing API test pattern.
- `./gradlew build` green, including Spotless, Checkstyle and ArchUnit module
  boundaries.

## Known trade-off

`overview()` iterates projects and fetches sources + recordings per project
(~2N reads for N projects). Acceptable at expected project counts for a
local-first simulator. A repository-level `COUNT` optimization is **not** done
here because `runtimeState` is not persisted (it is derived from the runtime
supervisor), so running/attention counts cannot be expressed as pure SQL counts;
splitting the logic would risk drift. Revisit only if project counts grow large.
